Architecture Summary

Goal: Transmit 160-byte media frames every 20ms across a hostile UDP network with strict bandwidth (≤ 2.0x) and deadline miss rate (≤ 1.0%) constraints, while minimizing playout delay.

1. The Problem Space

Using a traditional NACK-based ARQ strategy introduces a minimum delay penalty equal to the network's Round Trip Time (RTT). With network delays up to 80ms each way, ARQ would enforce a DELAY_MS of well over 160ms. To achieve the lowest possible playout delay, we must abandon feedback-driven retransmissions entirely and embrace Forward Error Correction (FEC).

2. Bandwidth Budgeting

The bandwidth limit dictates that total bytes traversing the relay cannot exceed 2.0x the raw stream size (which translates to 320 bytes allowed per 160-byte frame). A standard packet carrying a 4-byte sequence number and a 160-byte payload totals 164 bytes. This leaves exactly 156 bytes of budget per packet, making it mathematically impossible to include a full 160-byte FEC block in every packet.

Solution: We attach a 160-byte FEC block to 95% of transmitted packets. This yields an average packet size of 316 bytes, operating safely at 1.97x overhead.

3. The FEC Design: XOR(seq-1, seq-3)

Rather than simply duplicating the previous frame or XORing adjacent frames, the sender computes the parity block as Payload[seq - 1] XOR Payload[seq - 3].

By spacing the XOR parity across non-adjacent frames, the protocol achieves high resilience against independent drops as well as consecutive burst losses (up to length 2). If a burst loss wipes out both seq and seq+1, the receiver can still iteratively recover them both when seq+2 and seq+3 arrive.

4. Receiver Architecture

The harness player judges success based strictly on the first arrival time of a correct frame. Consequently, the receiver does not implement a playout scheduler or jitter buffer.

  • It utilizes a 256-element circular sliding window array (seen) to instantly discard duplicates and late arrivals.
  • Upon receipt of an FEC-enabled packet, it triggers an O(N) iterative recovery loop over the history window.
  • Recovered frames, along with naturally arriving frames, are blasted immediately to the player thread port, entirely removing local scheduling latency.

5. Results

When graded at a DELAY_MS of 140ms, this architecture seamlessly absorbs Profile B's 80ms network latency along with the maximum 60ms FEC reconstruction window, culminating in an overall frame miss rate of 0.40%.
